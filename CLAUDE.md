# CLAUDE.md — 개발 컨벤션 가이드

이 문서는 프로젝트에서 개발 시 지켜야 하는 기본적인 컨벤션을 정리한 문서입니다.
특정 언어나 프레임워크에 종속되지 않는 범용 원칙을 담고 있으며, 팀의 스택에 맞게 항목을 추가·수정해 사용하세요.

> 💡 이 파일은 Claude Code가 프로젝트 컨텍스트로 자동 참고합니다. 프로젝트 루트에 두면 작업 시 컨벤션이 자동으로 반영됩니다.

---

## 1. 기본 원칙

- **일관성 우선**: 개인 취향보다 팀에서 합의한 규칙을 우선한다. 기존 코드의 스타일을 따르는 것이 새 스타일을 도입하는 것보다 낫다.
- **가독성 우선**: 코드는 작성보다 읽히는 횟수가 훨씬 많다. 영리한 코드보다 명확한 코드를 지향한다.
- **작게, 자주**: 큰 변경 하나보다 작은 변경 여러 개로 나눈다. 리뷰와 롤백이 쉬워진다.
- **자동화 신뢰**: 포매터·린터·테스트 등 자동화 도구의 결과를 신뢰하고, 수동 규칙 준수에 의존하지 않는다.
- **문서화**: "왜(Why)"를 기록한다. "무엇(What)"은 코드가 말해주지만, 의도와 배경은 남기지 않으면 사라진다.

---

## 2. 네이밍 컨벤션

- **의미가 드러나는 이름**을 사용한다. `d`, `tmp`, `data2` 같은 모호한 이름을 피한다.
- 언어별 관례를 따른다: 변수·함수는 `camelCase` 또는 `snake_case`, 클래스·타입은 `PascalCase`, 상수는 `UPPER_SNAKE_CASE`.
- **불리언**은 `is`, `has`, `can`, `should` 접두사를 붙인다. (예: `isActive`, `hasPermission`)
- **함수는 동사**로, **변수·클래스는 명사**로 시작한다. (예: `getUser()`, `userRepository`)
- 약어는 팀에서 합의된 것만 사용하고, 도메인 용어는 통일한다. (예: `id` vs `identifier` 혼용 금지)
- 부정형 네이밍을 피한다. (`isNotValid` 대신 `isValid`)

---

## 3. 코드 스타일 / 포매팅

- **포매터를 강제**한다. (Prettier, Black, gofmt, spotless 등) 스타일 논쟁은 도구로 종결한다.
- 들여쓰기·따옴표·최대 줄 길이 등은 설정 파일로 고정하고 저장소에 커밋한다.
- **한 함수는 한 가지 일**만 한다. 함수가 길어지면(대략 40~50줄 초과) 분리를 검토한다.
- 매직 넘버·매직 문자열은 상수로 추출한다.
- 깊은 중첩을 피한다. 조기 반환(early return)으로 중첩을 줄인다.
- 주석은 코드로 표현할 수 없는 의도·제약·트레이드오프를 설명할 때만 사용한다. 코드를 그대로 옮긴 주석은 지양한다.

> 💡 포매팅·린트 규칙은 저장 시 자동 적용(format on save)과 커밋 전 훅으로 이중 보장하는 것을 권장합니다.

## 4. Git / 버전 관리

### 브랜치 전략

- `main`(또는 `master`)은 항상 배포 가능한 상태를 유지한다.
- 작업은 별도 브랜치에서 진행한다. 브랜치 이름은 `유형/설명` 형식을 권장한다.
  - 예: `feature/user-login`, `fix/token-expiry`, `refactor/order-service`, `chore/deps-bump`
- 브랜치는 오래 방치하지 않고 자주 통합한다.

### 커밋 메시지

- **Conventional Commits** 형식을 권장한다: `<type>(<scope>): <subject>`
  - `type`: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `perf`, `ci`
  - 예: `feat(auth): 소셜 로그인 추가`, `fix(cart): 수량 0 입력 시 오류 수정`
- 제목은 명령형·현재형으로, 50자 이내로 작성한다.
- 본문에는 "왜" 변경했는지, 부수 효과가 있는지 적는다.
- 하나의 커밋은 하나의 논리적 변경만 담는다.

### Pull Request

- PR은 작게 유지한다. (리뷰 부담이 큰 대형 PR 지양)
- 설명에 변경 목적, 주요 변경점, 테스트 방법, 스크린샷(UI인 경우)을 포함한다.
- 최소 1인 이상의 리뷰 승인 후 병합한다.
- CI(빌드·테스트·린트)가 통과해야 병합한다.

> 💡 민감 정보(API 키, 비밀번호, 토큰)는 절대 커밋하지 않습니다. `.gitignore`와 `.env`를 활용하고, 실수로 커밋했다면 키를 즉시 폐기·재발급하세요.

## 5. 코드 리뷰

- **비판이 아니라 코드에 대한 피드백**임을 전제로, 존중하는 태도로 리뷰한다.
- 리뷰 관점: 정확성, 가독성, 테스트 유무, 보안, 성능, 컨벤션 준수.
- 사소한 스타일은 자동화 도구에 맡기고, 리뷰는 설계와 로직에 집중한다.
- 요청·제안·필수 여부를 명확히 구분해 코멘트한다. (예: `nit:` 사소한 제안, `must:` 반드시 반영)
- 리뷰어와 작성자 모두 24시간 이내 응답을 지향한다.

---

## 6. 테스트

- 새 기능·버그 수정에는 테스트를 함께 추가한다.
- **테스트 피라미드**를 따른다: 단위 테스트(다수) → 통합 테스트(적당히) → E2E(소수).
- 테스트는 독립적이고 반복 가능해야 한다. 실행 순서나 외부 상태에 의존하지 않는다.
- 테스트 이름은 "무엇을, 어떤 조건에서, 어떤 결과"가 드러나게 작성한다.
- 커버리지 수치보다 **중요한 경로가 검증되는지**를 우선한다.
- 실패하는 테스트를 방치하거나 무력화한 채 병합하지 않는다.

---

## 7. 에러 처리 / 로깅

- 예외를 조용히 삼키지 않는다. 무시할 경우 명확한 이유를 주석으로 남긴다.
- 사용자에게는 친절한 메시지를, 로그에는 원인 추적에 필요한 상세 정보를 남긴다.
- 로그 레벨(`debug` / `info` / `warn` / `error`)을 상황에 맞게 사용한다.
- 로그에 민감 정보(비밀번호, 토큰, 개인정보)를 남기지 않는다.
- 실패는 빠르게 드러낸다(fail fast). 잘못된 상태를 계속 진행시키지 않는다.

---

## 8. 보안

- 모든 외부 입력은 신뢰하지 않고 검증·이스케이프한다. (SQL 인젝션, XSS 등 방지)
- 인증·인가는 서버 측에서 반드시 재검증한다.
- 비밀 값은 코드가 아닌 환경 변수·비밀 관리 도구에 둔다.
- 의존성은 정기적으로 업데이트하고, 알려진 취약점을 점검한다.
- 최소 권한 원칙을 지킨다. 필요 이상의 접근 권한을 부여하지 않는다.

---

## 9. 의존성 / 프로젝트 구조

- 의존성 추가는 신중히 결정한다. 유지보수 상태·라이선스·번들 크기를 확인한다.
- 버전을 잠금(lock) 파일로 고정하고 저장소에 커밋한다.
- 폴더 구조는 도메인·기능 단위로 응집도 있게 구성한다.
- 설정·상수·환경별 값은 한 곳에서 관리한다.
- 사용하지 않는 코드·의존성은 주기적으로 제거한다.

---

## 10. 문서화

- `README`에는 프로젝트 개요, 설치·실행 방법, 주요 스크립트를 기록한다.
- 주요 아키텍처 결정은 ADR(Architecture Decision Record) 등으로 남긴다.
- 공개 API·함수에는 입력·출력·예외를 명시한다.
- 문서는 코드와 함께 최신 상태로 유지한다. 오래된 문서는 잘못된 문서보다 위험하다.

## 11. 작업 진행 원칙 (Cowork/AI 협업 시)

> 💡 이 프로젝트에서 AI 어시스턴트와 협업할 때는 다음을 준수합니다.

- **파일 수정 전 반드시 작업 계획을 먼저 보고**한다.
- **파일 영구 삭제는 명시적 허가 없이 절대 실행하지 않는다.**
- **대규모 파일 작업 전에는 수동 백업을 권장**하는 안내를 포함한다.
- 설정 경로는 항상 `Settings → 메뉴 → 하위메뉴` 형식으로 표기한다.
- 주의사항은 💡 아이콘과 함께 별도로 강조한다.
