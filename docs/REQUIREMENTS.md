# AI 수업 자료·결과물 공유 앱 — 요구사항 정의서

오프라인 AI 수업에서 (1) 강사가 회차별 강의 자료를 공유하고 (2) 수강생이 본인이 만든 웹사이트·작업물을 올려 서로 구경·피드백하는 웹앱.
랜딩 페이지 없음. 접속 즉시 기능 화면. 웹 우선, 모바일 대응.

---

## 1. 인터뷰로 확정된 요구사항

| # | 항목 | 결정 |
|---|---|---|
| 1 | 접근 | **로그인 없음** — URL을 아는 사람 누구나 |
| 2 | 클래스 구조 | 단일 클래스, **회차(주차)별 정리** |

| 4 | 결과물 공유 | **카드 그리드 갤러리** |

| 8 | 썸네일 | **OG 이미지 자동** → 실패 시 직접 업로드 |

| 12 | 디자인 | 깔끔한 미니멀 (흑백 + 포인트 컬러 1개) |

## 2. 핵심 설계 결정 (먼저 읽어주세요)

요구사항 1의 "로그인 없음"과 요구사항 10의 "무료 티어"가 나머지 전부를 제약합니다. 그 결과 내려진 결정들입니다.

### 2.1 검색엔진에 잡히지 않게 기본 설정하고, 게시 시 공개 동의를 받는다
로그인이 없으니 URL이 단톡방을 거쳐 퍼지는 건 시간 문제입니다. 한 번 구글에 색인되면 사이트에서 지워도 캐시와 아카이브에 남습니다. 특히 코딩 수업에서는 학생이 올린 스크린샷에 API 키나 실명이 함께 찍히는 사고가 흔합니다.
→ 전역 `X-Robots-Tag: noindex, nofollow` + robots.txt Disallow. 게시 폼에 "링크를 아는 누구나 볼 수 있어요" 고지 + 필수 체크박스.
**지금 막는 건 설정 한 줄이고, 나중에 공개로 바꾸는 건 언제든 가능하지만 반대는 불가능합니다.**

### 2.2 브라우저에서 Supabase를 직접 호출하지 않는다 — 모든 읽기·쓰기는 서버 경유
가장 위험한 건 "로그인 없음"이 아니라 **로그인 없음 + 브라우저에서 Supabase 직접 호출**의 조합입니다. devtools 여는 법을 배우는 수업에서, RLS를 느슨하게 열어두면 수강생이 콘솔 한 줄로 갤러리 전체를 지울 수 있습니다. 5주차쯤 호기심 많은 수강생이 "되나?" 하고 눌러보는 순간 8주치 결과물이 사라집니다.
→ anon key를 클라이언트에 아예 내려보내지 않음. 모든 테이블 RLS 전면 거부(정책 0개). service_role key는 `import 'server-only'` 모듈에만 — 클라이언트 번들에 섞이면 빌드가 실패합니다.
→ **"쓰기는 전부 서버 라우트"를 1일차 규칙으로 못박고 시작.** 나중에 바꾸면 전면 재작업입니다.

### 2.3 관리자 비밀번호는 프로젝터에 연결된 상태에서 입력하지 않는다
가장 현실적인 유출 경로는 해킹이 아니라 강의실 화면에 비밀번호를 타이핑하는 순간입니다. 20명이 보고 있고 그중 누군가는 기억합니다.
→ 세션은 기본 12시간, `[이 기기를 30일 기억 — 강의실·공용 PC에서는 체크하지 마세요]` 체크 시 30일. 관리 화면에 활성 세션 목록 + **[모든 기기 로그아웃]**. 비밀번호 변경 시 전 세션 자동 무효화.
→ 비밀번호는 DB에 argon2id + 페퍼로 저장(환경변수 아님). 재배포 없이 바꿀 수 있어야 실제로 바꾸게 됩니다.

### 2.4 아무것도 진짜로 지워지지 않는다 (소프트 삭제)
Supabase 무료 플랜에는 사용자가 복원할 수 있는 백업이나 PITR이 **없습니다**. 잘못 누른 삭제가 곧 영구 손실입니다. 3주차 자료가 통째로 날아가면 그 뒤로 이 도구를 신뢰하지 않고 다시 카톡으로 자료를 뿌리게 됩니다.
→ 모든 삭제는 `deleted_at`. 물리 CASCADE 금지(`materials` FK는 `ON DELETE RESTRICT`). 회차 삭제는 **회차 제목을 직접 타이핑**해야 확인.
→ 회차를 지워도 그 회차의 결과물은 지우지 않고 '회차 없음'으로 표시(복구하면 되돌아옴).
→ 단, **이미지 파일 자체는 삭제 요청 시 즉시 물리 삭제** — 화면에서만 사라지고 파일 주소가 살아 있으면 "지워주세요" 요청의 의미가 없습니다.

### 2.5 사진은 올리는 순간 브라우저에서 자동으로 줄여서 저장한다
이 하나가 네 가지를 동시에 해결합니다.
- 폰 사진의 **GPS EXIF 제거** (집에서 찍었다면 집 주소가 공개 웹에 올라갑니다)
- 아이폰 **HEIC → JPEG** (올린 본인 사파리에선 잘 보이고 노트북 크롬에서만 깨져서, 정작 올린 사람이 문제를 모릅니다)
- "다 같이 갤러리 열어보세요" 할 때 **강의실 와이파이 마비 방지** (20명 × 40~60장 × 3~8MB = 100MB 이상이 한꺼번에 몰림)
- 무료 **Storage/egress 용량** 유지

→ 클라이언트 canvas로 긴 변 1600px 원본 + 400px 썸네일 2장 생성 후 업로드. 용량이 20분의 1이 됩니다.
→ 단, 클라이언트 리사이즈는 **보안 수단이 아닙니다.** 서버에서 매직바이트 검증 + SVG/HTML 전면 거부를 별도로 겁니다.
→ 나중에 넣으면 이미 올라간 원본을 소급 처리해야 합니다.

### 2.6 링크 썸네일은 "실패가 정상"이라고 보고, 실패 화면을 먼저 만든다
이 수업이 실제로 만들어낼 링크를 하나씩 보면 — 처음 배포한 Vite/정적 사이트는 OG 태그가 아예 없고, Colab은 전부 똑같은 제네릭 로고를 주고(기술적으론 '성공'인데 20장이 같은 그림), Figma·Notion은 로그인 벽 뒤, 인스타는 비로그인 fetch에 로그인 벽을 돌려줍니다. **절반 이상이 쓸만한 썸네일을 못 얻습니다.**
→ 실패를 '깨짐'이 아니라 '의도된 디자인'으로: URL 해시 기반 결정론적 그라데이션 + 도메인명 카드(미니멀 톤과 잘 맞습니다).
→ 게시 시점에 동기로(5초 타임아웃) 추출해 결과를 즉시 보여주고, 실패하면 **그 자리에서** 직접 업로드를 유도. 나중에 고치라고 하면 아무도 안 고칩니다.
→ 성공한 og:image는 핫링크하지 않고 즉시 우리 Storage로 복사(방문자 IP가 외부로 새는 것 + 나중에 이미지가 바뀌는 것 차단).

### 2.7 학생이 PIN을 잊는 것은 예외가 아니라 기본값으로 가정한다
3주 전에 정한 4자리를 기억하는 사람은 거의 없습니다. 대비가 없으면 학생은 수정 대신 새 글을 올리고, 갤러리에 같은 프로젝트가 두세 개씩 쌓입니다.
→ 같은 브라우저에서는 PIN을 아예 묻지 않음(기기 기억 기본 ON). 실제 수정의 대부분은 '올린 직후 오타 고치기'라 PIN 쓸 일 자체가 거의 사라집니다.
→ 게시 직전 `"PIN 1234로 등록됩니다. 맞나요?"` 를 마스킹 해제로 1회 확인(오타 방지, 입력 마찰은 늘지 않음).
→ 잊었을 때는 강사가 대신 삭제. PIN은 5회 실패 시 15분 잠금 + IP 레이트리밋 + 해시 저장(4자리는 1만 가지라 lockout 없으면 17분에 뚫립니다).

### 2.8 회차 번호는 한 번 쓰면 재사용하지 않는다
주소가 `/weeks/3` 형태라 단톡방에 뿌린 링크가 오래 살아 있습니다. 번호를 재사용하면 예전에 공유한 '3주차 자료' 링크가 어느 날 전혀 다른 회차를 가리킵니다. 번호가 중간에 비는 건 눈에 띄지만, 링크가 조용히 다른 곳을 가리키는 것보다 훨씬 낫습니다.

## 3. 화면 구조

| 라우트 | 화면 | 역할 |
|---|---|---|
| `/` | 홈 (회차 허브) | 공지 배너 + 이번 주차 하이라이트 + 회차 그리드 + 최근 결과물 8개 |
| `/weeks/[order_no]` | 회차 상세 | 자료 리스트(파일·링크 통합) + 그 회차 결과물 섹션 |
| `/gallery` | 결과물 갤러리 | 카드 그리드 정본. 회차·태그 필터/검색/정렬, 24개씩 [더 보기] |
| `/gallery/[id]` | 결과물 상세 | 큰 이미지 또는 링크 + 이모지 반응 + 댓글 |
| `/gallery/new` | 결과물 올리기 | 링크/이미지 토글 + 제목·설명·회차·태그 + 닉네임·PIN |
| `/gallery/[id]/edit` | 결과물 수정 | PIN 게이트(같은 기기면 자동 통과) |
| `/qr` | QR 접속 화면 | 프로젝터용 풀스크린. QR 대상 전환(홈/이번 회차/갤러리/올리기) |
| `/admin` | 관리자 게이트 + 대시보드 | 비번 입력 → 관리자 모드 ON. 공지 편집, 회차 관리, 스토리지 사용량 |
| `/admin/gallery` | 게시물 관리 | PIN 없이 강제 삭제, 회차 변경, 댓글 관리 |

**핵심 IA 결정**
- 강사와 수강생이 **같은 URL**로 들어옵니다. 관리자 모드가 켜지면 편집 컨트롤만 인라인으로 더 보입니다(별도 관리자 사이트 없음).
- 비관리자에게는 헤더 맨 끝 회색 12px `[관리자]` 텍스트 링크 하나만 노출.
- 갤러리는 `/gallery`가 정본이고, 회차 상세의 결과물 섹션은 그 필터된 뷰입니다.
- 필터 상태는 전부 URL 쿼리에 반영 (`/gallery?week=3&tag=랜딩페이지&sort=hot`) — 새로고침·단톡방 공유에도 같은 화면.

**모바일**
- 하단 고정 탭바 `[홈][갤러리]`, 갤러리 2열, 필터는 바텀시트, 우하단 FAB `[+]`.
- 게시 폼의 타입 토글 기본값이 폰에서는 `[이미지]`로 뒤집힘(폰은 사진 업로드가 주 용도).
- 결과물 상세는 모달 대신 전체 페이지, 좌우 스와이프로 이전/다음.
- PIN 입력은 `inputmode="numeric"`, 화면 상단 1/3에 배치(키보드에 가리지 않게).

## 4. 데이터 모델

| 테이블 | 역할 |
|---|---|
| `class_sessions` | 회차. `order_no`는 삭제분 포함 **전역 유니크**(재사용 금지). 순서 변경 대비 `DEFERRABLE` 인덱스 |
| `materials` | 강의 자료. `kind = file \| link` 단일 테이블 (정렬을 한 리스트로 관리) |
| `gallery_posts` | 결과물. `kind = link \| image`, `pin_hash`, `visitor_id`, `tags[]` |
| `gallery_posts_public` | **뷰**. `pin_hash`/`visitor_id`/`created_ip_hash` 제외. 읽기는 항상 이 뷰만 |
| `comments` | 댓글. PIN 없음 — `visitor_id` 쿠키 소유권 + 운영자 모더레이션 |
| `reactions` | 이모지 반응. PK `(post, visitor_id, emoji)`. **IP를 주체로 쓰면 안 됨** (같은 강의실 NAT에서 서로의 반응을 막음) |
| `announcements` | 공지 배너. 이력 보존 위해 다행 + 활성 1건 |
| `link_previews` | OG 캐시. URL당 1행 dedupe → 외부 fetch 횟수(SSRF 표면)와 용량 동시 감소 |
| `admin_credential` | 비번 해시 1행 + `password_version` |
| `admin_sessions` | 불투명 토큰 세션 (원격 폐기 가능해야 하므로 JWT 아님) |
| `rate_limits` | 고정 윈도우 카운터. 서버리스라 인메모리는 무의미 → Postgres 테이블 |
| `audit_log` | '누가 지웠나'를 사후 확인할 유일한 수단 + PIN 무차별 대입 탐지 |
| `upload_intents` | 서명 URL 발급 기록. 커밋 단계에서 임의 경로 주장을 걸러내는 근거 |
| `site_settings` | 클래스명, QR 기준 URL, `archive_mode` |

**날짜**: 서버 런타임이 UTC이므로 모든 포매팅에 `Asia/Seoul` 고정(유틸 하나로 강제). 안 하면 밤 9시 이후 게시물 날짜가 하루 밀립니다.

## 5. 보안 대응 (전부 P0)

| 위험 | 대응 |
|---|---|
| 갤러리 전체 삭제 | 전 테이블 RLS 기본 거부, 쓰기는 서버 라우트만, service_role은 `server-only` 모듈 |
| 4자리 PIN 무차별 대입 | argon2 해시 + 공개 뷰에서 컬럼 제외 + 5회 실패 15분 잠금 + IP 레이트리밋 + 취약 PIN(1234, 생일 등) 거부 |
| **파일 업로드 4.5MB 한계** | Vercel 함수 바디 상한 때문에 프록시 방식은 **처음부터 불가능**. `createSignedUploadUrl` → 브라우저가 Storage로 직접 PUT → 서버가 커밋 시 검증(앞 4KB만 Range로 받아 매직바이트 판별) |
| SSRF (OG 추출) | 외부 fetch는 `/api/links/preview` 한 곳만. http/https + 80/443만, DNS 해석 후 사설·루프백·링크로컬 거부, 리다이렉트 2회 & 매 홉 재검증, 타임아웃 5초, 본문 1MB 상한. 실패 시 원본 응답을 에러 메시지에 절대 반영 안 함 |
| 업로드 파일 XSS | SVG·HTML 전면 거부. Storage가 별도 오리진(`*.supabase.co`)이라 실행돼도 앱 쿠키 접근 불가 — **이 오리진 분리는 유지할 가치가 있으므로 커스텀 도메인을 앱 서브도메인으로 붙이지 말 것** |
| 스팸 | 레이트리밋 + 허니팟 + 폼 렌더 후 3초 검사 (공개 URL의 무인증 POST를 무방비로 여는 건 며칠 단위 문제) |
| 중복 제출 | 클라이언트가 폼 진입 시 생성한 uuid를 그대로 PK로 보내고 `ON CONFLICT DO NOTHING` |
| 백업 없음 | GitHub Actions 크론으로 주 1회 `pg_dump` → private 저장소. **수업 전 1회는 복원까지 실제로 해볼 것** — 복원해 본 적 없는 백업은 백업이 아닙니다 |

## 6. 범위

### P0 — 첫 배포
1. 공개 접근 + 앱 셸 (shadcn 기본 토큰 + 색 3개, 커스텀 디자인 없음)
2. 회차 구조: 생성 / 목록 / 상세
3. 강의 자료 등록(파일 + 링크) + 삭제 — **제품 목적 1 완성**
4. 운영자 비밀번호 게이트 (자료 업로드보다 **먼저** 만들어 쓰기 경로를 처음부터 가드 뒤에)
5. 갤러리 카드 그리드 + 게시(이미지 **1장 고정**) — **제품 목적 2 완성**
6. 닉네임 + PIN 저장 / PIN으로 본인 글 **삭제**
7. **클라이언트 이미지 리사이즈** + 서버 매직바이트 검증
8. 썸네일 **폴백 디자인**(그라데이션 카드) + 직접 업로드
9. 태그 **저장**(입력·스키마·GIN 인덱스) — 필터 UI는 P1이지만 저장은 첫날부터
10. 커서 페이지네이션 (`/api/gallery`, 24개씩 [더 보기])
11. noindex + 게시 공개 동의 체크박스
12. 레이트리밋 + 허니팟
13. 공지 배너 + QR 화면
14. 모바일 반응형(동작 수준) + `not-found.tsx` / `error.tsx` 한국어
15. 푸터에 삭제 요청 안내 + 강사 연락 수단

### P1 — 2~3주차까지
이모지 반응(1시간 작업, **2회차 전 권장**) · 댓글 · 회차/태그 필터 UI · 검색 · OG 자동 추출 · PIN으로 **수정** · 서버 sharp 재인코딩 · 회차 삭제 & 자료 순서 변경 · 갤러리 상세 모달(인터셉팅 라우트) · 다중 이미지 · `archive_mode`

### P2 — 필요해지면
정렬 옵션/조회수 · 다중 이미지 캐러셀 · 이미지 최적화 파이프라인 고도화

### 만들지 않는 것
계정 시스템 일체 · 여러 기수/멀티테넌시 · 알림(메일·카톡·슬랙) · 파일 버전관리·폴더트리·PDF 내장 뷰어 · 통계 대시보드 · 신고/모더레이션 큐/욕설 필터/Turnstile · Realtime 구독 · 무한 스크롤 · 다크모드 · i18n · SEO/OG 메타(외부 유입 목적이 아님) · 리치 텍스트 에디터 · 관리자 계정 다중화

> **범위 조정 근거**: 요구사항 7(댓글·검색·필터)을 전부 P1로 미룬 이유 — 첫 수업에는 결과물이 0~5개라 검색·필터할 대상이 없고, 피드백은 그 자리에서 말로 오갑니다. 반면 댓글은 작성자 식별·삭제 권한·스팸 대응까지 딸려와 P0 두 개분의 비용입니다. **단 태그 저장만은 첫날부터** — 나중에 켜면 그때까지 올라간 결과물의 태그가 비어 필터가 무용지물이 됩니다.

## 7. 구현 순서

1. **배포 파이프라인 먼저** — 빈 페이지라도 실제 URL로 배포까지 성공시킨다. 마지막 날 배포에서 막히는 게 최대 리스크이고, QR이 가리킬 주소도 여기서 확정된다.
2. **관리자 시드 스크립트** (`pnpm seed:admin`) — 이게 없으면 `admin_credential`이 비어 `/admin`을 통과할 수 없고, 배포에 성공해도 앱을 쓸 수 없는 상태로 시작한다.
3. 데이터 모델 + Storage 버킷 확정
4. 운영자 비밀번호 게이트 (쓰기 공통 가드)
5. 회차 생성 + 목록/상세
6. 자료 업로드(서명 URL) + 링크 등록 + 삭제 → **여기서 한 번 배포해 실제 PDF를 폰으로 열어본다**
7. 갤러리 게시 폼 + 카드 그리드 + PIN 삭제
8. 공지 배너 + QR 화면
9. **테스트** — 유닛: ① 운영자 세션 가드(만료·폐기·`password_version` 불일치) ② PIN 해싱·검증·잠금 카운터 ③ SSRF URL 검증 유틸(사설 대역, IPv4-mapped IPv6 `::ffff:10.0.0.1`, 리다이렉트 홉 재검증) ④ 취약 PIN 거부. e2e: `관리자 로그인 → 자료 업로드 → 갤러리 게시 → PIN 삭제` 해피패스 1개
10. 실제 폰으로 QR 스캔 → 자료 열람 → 이미지 업로드 리허설
11. 1주차 실데이터 투입 (수업 **하루 전**까지)

## 8. 착수 전 확정 항목 — 확정 완료

| # | 항목 | 결정 | 영향 |
|---|---|---|---|
| 1 | 수업 성격 | **무료 / 사내 스터디** | Vercel Hobby 무료 플랜 그대로 사용. 약관 문제 없음, 운영비 0원 |
| 2 | 도메인 | **`*.vercel.app` 서브도메인** | 커스텀 도메인 구매·연결 단계 없음. 1단계 배포 시 확정되는 주소를 QR 기준(`site_settings.base_url`)으로 고정 |
| 3 | 강의 슬라이드 | **HTML 슬라이드를 배포하고 그 주소를 링크로 등록** | 추가 개발 없음 — 이미 설계된 `materials.kind = 'link'`로 커버. **HTML/SVG 업로드 거부 규칙은 그대로 유지** |
| 4 | 첫 수업 날짜 | 미정 | 확정되면 P0에서 더 덜어낼지 재검토 |

> **슬라이드가 링크 방식이라 부수적으로 좋아진 점**: 파일 업로드 경로로 들어오는 건 PDF·이미지 정도가 되어 무료 Storage 1GB 압박이 사실상 사라집니다. 다만 Vercel 함수 4.5MB 한계는 여전히 유효하므로 서명 URL 방식(5장)은 그대로 갑니다.
>
> **링크 방식의 대가**: 슬라이드를 고칠 때마다 별도로 재배포해야 하고, 배포 주소가 죽으면 자료도 함께 죽습니다. 회차 상세의 링크 항목에 [열기] 실패를 감지할 방법이 없으므로, 학기 말에 한 번 링크 상태를 점검하는 것을 권합니다.

## 9. 총평

네 갈래로 독립 설계한 결과가 서로 일관되게 같은 결론(서버 경유 쓰기, 소프트 삭제, 이미지 재인코딩, SSRF 격리)에 도달했습니다. 착수 가능한 상태입니다.

**"소규모라 괜찮다"가 실제로 성립하는 항목**은 동시접속(20명은 Vercel에 아무것도 아님)과 DB 용량(500MB 근접 불가)뿐입니다. **egress·스토리지·스팸·프라이버시는 규모와 무관하게 터집니다.**

**절대 나중으로 미루면 안 되는 4가지** — 전부 나중에 넣으면 전면 재작업이거나 이미 벌어진 사고를 되돌릴 수 없습니다:
1. 쓰기 경로를 처음부터 서버 뒤에 두는 구조
2. 소프트 삭제
3. noindex
4. 클라이언트 이미지 리사이즈
